## General Sub-Agent Guidelines

1. Read the skill SKILL.md first.
2. `cd` to `{target_path}` (cross-repo context).
3. Invoke SDD scripts via the shim — see
   [`tool-patterns.md` § Invocation](tool-patterns.md#invocation) for the
   canonical rule, failure-recovery, and the single-repo fallback.
   Cross-repo sub-agents add two adjustments on top:
   - Wrap every call with a guard `cd` (sub-agents may not persist CWD):
     ```
     cd {target_path} && .spec-workflow/sdd {group}/{script}.py [args...]
     ```
   - When the shim is unavailable, fall back to the Layer 3 form
     documented in
     [`bootstrap-pattern.md`](bootstrap-pattern.md), passing
     `--project {target_path}` so cross-repo paths resolve.
4. Follow all steps in the skill. Do NOT skip steps.
5. Return structured results in the format specified by each template below.

## Scope Variants

The Review and Approval Pipeline invokes review sub-agents at two scopes:

- **`per-document`**: `doc_list` contains a single document. The sub-agent
  reviews only that document (skip cross-document validation).
- **`final`**: `doc_list` contains all documents. The sub-agent runs the full
  review including cross-document consistency checks.

## Headline Narrative

Every reviewed-doc reply MUST start with a single-line headline that
echoes the canonical shape (defined verbatim in
`scripts/review/pipeline_phases/templates.py::OVERALL_STATUS_NARRATIVE_TEMPLATE`):

```
Reviewed-docs status: <PASS|NEEDS_WORK|FAIL> (Tier 1 facets: <summary>; Tier 2 score: <n>/<max>)
```

The post-review verifier compares the substituted `overall_status`
against the artifact's authoritative status. A disagreement surfaces
as a `sub_agent_verdict_mismatch` finding with
`direction == "narrative_headline_drift"` — observability only; the
artifact remains the routing source of truth.

The writer captures the headline under `sub_agent_narrative_headline`
in the review-quality artifact so the verifier can read it back; an
artifact missing the field stays valid (older reviews pass through).

## Spec Docs Review Template

For `sdd-review-spec-docs`:

```
You are reviewing spec documents for quality. Follow the SDD review
skill workflow exactly.

Read the skill file FIRST:
  {review_skill_path}

Parameters:
  - Spec name: {target_name}
  - Project path: {target_path}
  - Available documents: {doc_list}

All SDD script invocations MUST use the shim runner:
  cd {target_path} && .spec-workflow/sdd {group}/{script}.py [args...]

Follow all steps in the skill.

Verification: After completion, confirm that the file
  {target_path}/.spec-workflow/specs/{target_name}/review-quality.json
exists.

Return **two distinct status fields**:

- **Reviewed-docs status** (drives fix-loop routing): PASS /
  NEEDS_WORK / FAIL computed over the docs examined in this gate.
- **Artifact completeness** (informational): INCOMPLETE when any
  expected doc is missing; PASS once every expected doc exists.
  Never drives fix-loop routing.

Plus: overall score, per-facet results (facet name + score + pass/fail),
issues list with severities, and confirmation of artifact creation.
```

## Code Review Template

For `sdd-review-code`:

```
You are reviewing code for quality. Follow the SDD code review
skill workflow exactly.

Read the skill file FIRST:
  {review_skill_path}

Parameters:
  - Mode: {mode} (spec-aware | standalone)
  - Spec name: {target_name} (required when spec-aware; omit for standalone)
  - Project path: {target_path}
  - Changed files: {changed_files} (optional — auto-detected from git if omitted)

All SDD script invocations MUST use the shim runner:
  cd {target_path} && .spec-workflow/sdd {group}/{script}.py [args...]

Follow all steps in the skill. When in standalone mode, skip spec-only steps.

Return **two distinct status fields**:

- **Reviewed-docs status** (drives fix-loop routing): PASS /
  NEEDS_WORK / FAIL computed over the dimensions examined.
- **Artifact completeness** (informational): INCOMPLETE when any
  expected dimension is missing; PASS once every expected dimension
  has been recorded. Never drives fix-loop routing.

Plus: overall score, per-dimension results (dimension name + score + pass/fail),
findings list with severities (file, line, description, recommendation),
positive observations, and executive summary.
```

## PRD Docs Review Template

For `sdd-review-prd`:

```
You are reviewing a PRD for quality. Follow the SDD review
skill workflow exactly.

Read the skill file FIRST:
  {review_skill_path}

Parameters:
  - Feature name: {target_name}
  - Project path: {target_path}
  - Available documents: {doc_list}

PRD is located via discovery manifest, NOT spec status:
  1. Read manifest at {target_path}/.spec-workflow/discovery/{target_name}/manifest.json
  2. Filter artifacts where type == "prd"
  3. PRD file path: {target_path}/.spec-workflow/discovery/{target_name}/{prd_name}

Do NOT use spec/check-status.py for PRD discovery — it only searches .spec-workflow/specs/.

All SDD script invocations MUST use the shim runner:
  cd {target_path} && .spec-workflow/sdd {group}/{script}.py [args...]

Follow all steps in the skill.

Verification: After completion, confirm that the file
  {target_path}/.spec-workflow/discovery/{target_name}/review-quality.json
exists.

Return **two distinct status fields**:

- **Reviewed-docs status** (drives fix-loop routing): PASS /
  NEEDS_WORK / FAIL computed over the docs examined in this gate.
- **Artifact completeness** (informational): INCOMPLETE when any
  expected doc is missing; PASS once every expected doc exists.
  Never drives fix-loop routing.

Plus: overall score, per-facet results (facet name + score + pass/fail),
issues list with severities, anti-pattern detections, and confirmation
of artifact creation.
```
